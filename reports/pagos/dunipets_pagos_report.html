Recomendación de Pasarelas

Estrategia por fases para DuniPets — desde lanzamiento hasta escala

Recomendación Principal

ePayco — Pasarela colombiana

ePayco es la opción más completa para DuniPets en su etapa actual por una razón estructural: es la única pasarela colombiana que ofrece split payment automático dentro de la misma transacción sin requerir que los cuidadores abran cuentas en un servicio externo. Cuando un dueño paga $50,000 COP, ePayco distribuye automáticamente $40,000 al cuidador y $10,000 a DuniPets en el mismo evento de pago.

CriterioEstado
Persona natural sin Cámara de Comercio✅ Confirmado
Link de pago sin desarrollo técnico✅ Disponible
Daviplata nativo (20M+ usuarios Colombia)✅ Integración nativa
Split automático para pagar cuidadores✅ Sin costo adicional
Comisión competitiva✅ 2.68–2.99% + $900
Documentación técnica✅ docs.epayco.com
Empresa 100% colombiana✅ Medellín
⚠️
Limitación principal Nequi no es integración nativa directa (funciona via PSE). La tarifa fija de $900 COP es alta para tickets de $13,500 COP (~6.7%). Mitigación: establecer ticket mínimo de $25,000 COP.
Alternativa / Complementaria

Wompi (Bancolombia) — si los fundadores tienen cuenta Bancolombia

  • ✅ Nequi nativo (Bancolombia es propietario de Nequi) — dispersión en segundos
  • ✅ Comisión PSE más baja del mercado (~1.49% + $1,200 vs $2,000 plano de ePayco)
  • ✅ Wompi Payouts: paga a cualquier banco colombiano + Nequi en un solo batch
  • ✅ Red de 17,600 corresponsales para efectivo — mayor cobertura
  • ❌ Requiere cuenta corriente o de ahorros Bancolombia para recibir desembolsos
  • ⚠️ Payouts son proceso separado del cobro — planes desde $210,000 COP/mes o $1,849 + 0.4% por pago

Uso complementario: Si ePayco es la pasarela principal de cobro, Wompi Payouts puede usarse exclusivamente para dispersar a cuidadores con cuenta Nequi.

  • Split 80% sitter / 20% DuniPets por transacción
  • Wompi Payouts para cuidadores con Nequi
  • Webhook para notificaciones automáticas
  • Antifraude ePayco nativo (incluido)

Año 2+ · SAS registrada · >$100M COP/mes

  • Wompi Gateway (plan negociado con Bancolombia)
  • Wompi Payouts Premium (volumen justifica plan mensual)
  • PlaceToPay / Kushki como alternativa enterprise
  • Cuotas sin interés para tickets premium
  • SLA garantizado y ejecutivo comercial dedicado

Árbol de decisión rápida

¿Los fundadores tienen cuenta Bancolombia? ├── SÍ → Wompi como pasarela única (cobro + dispersión en un proveedor) └── NO → ePayco como pasarela principal ├── Cuidadores cobran en banco o Daviplata → ePayco Pagos Divididos (Fase 2) └── Cuidadores prefieren Nequi → ePayco (cobro) + Wompi Payouts (dispersión Nequi)

Riesgos y limitaciones

Riesgo 1 — Tarifa fija alta en tickets bajos

La tarifa fija de $900 COP de ePayco representa el 6.7% de un ticket de $13,500 COP, llevando la comisión total a ~10%.

Establecer ticket mínimo de $25,000 COP en la plataforma.

Riesgo 2 — Wompi: dependencia de Bancolombia

Si los fundadores no tienen cuenta Bancolombia, el modelo Agregador de Wompi queda bloqueado para recibir pagos.

Abrir cuenta Bancolombia antes de integrar Wompi, o usar ePayco que acepta cualquier banco.

Riesgo 3 — PSE de ePayco en tickets <$60K COP

$2,000 COP plano es costoso para un servicio de $20,000 COP (10% solo por PSE).

Incentivar pago con tarjeta o establecer cobro adicional visible para PSE.

Riesgo 4 — Registro DIAN / facturación

Al superar ciertos umbrales, ePayco puede requerir NIT activo y bloquear la cuenta.

Registrar la SAS antes de superar $4,800 UVT anuales (~$228M COP en 2026).

Riesgo 5 — PayU / Rapyd transición

PayU fue adquirida por Rapyd. La continuidad de contratos y soporte en Colombia no está garantizada.

No integrar PayU como proveedor principal en 2026.

Riesgo 6 — Adopción de Nequi por cuidadores

ePayco no tiene dispersión nativa a Nequi. Cuidadores que prefieran Nequi requieren proceso adicional.

Fase 1: pagar desde Nequi personal DuniPets. Fase 2: Wompi Payouts como canal Nequi.

Paso 2 — Backend: crear token de pago con split

El split se define en el backend para evitar manipulación desde el cliente. Endpoint: POST https://secure.epayco.co/payment/process

Python
import requests
import hashlib

def crear_pago_con_split(reserva_id, monto_total_cop, sitter_id):
    """
    monto_total_cop: int, ej. 50000
    sitter_id: str, ID del receptor registrado en ePayco
    """
    # Firma de seguridad (SHA256)
    firma = hashlib.sha256(
        f"{P_CUST_ID}^{P_KEY}^{reserva_id}^{monto_total_cop}^COP".encode()
    ).hexdigest()

    payload = {
        "P_CUST_ID_CLIENTE": P_CUST_ID,
        "P_KEY": P_KEY,
        "x_ref_payco": reserva_id,          # ID único de la reserva
        "x_amount": monto_total_cop,
        "x_currency_code": "COP",
        "x_description": f"Reserva DuniPets #{reserva_id}",
        "x_signature": firma,
        "x_redirect_url": "https://dunipets.co/reserva/confirmacion",
        "x_response_url": "https://dunipets.co/reserva/respuesta",

        # Pagos Divididos — Tipo 02 = porcentaje
        "split": [
            {
                "id_receiver": sitter_id,    # ID del cuidador en ePayco
                "type": "02",                # Tipo 02: porcentaje
                "value": "80"               # 80% va al cuidador
            }
            # El 20% restante queda en la cuenta principal DuniPets
        ]
    }

    response = requests.post(
        "https://secure.epayco.co/payment/process",
        json=payload,
        headers={"Content-Type": "application/json"}
    )
    return response.json()

Paso 4 — Backend: recibir webhook de confirmación

Python (FastAPI)
from fastapi import FastAPI, Request
import hashlib

app = FastAPI()

@app.post("/api/webhook/epayco")
async def webhook_epayco(request: Request):
    data = await request.json()

    # 1. Validar firma — NUNCA omitir esta validación
    firma_esperada = hashlib.sha256(
        f"{P_CUST_ID}^{P_KEY}^{data['x_ref_payco']}^{data['x_amount']}^{data['x_currency_code']}^{data['x_transaction_state']}".encode()
    ).hexdigest()

    if firma_esperada != data.get('x_signature'):
        return {"error": "Firma inválida"}, 400

    estado = data['x_transaction_state']
    reserva_id = data['x_ref_payco']

    if estado == "Aceptada":
        await actualizar_reserva(reserva_id, estado="pagada")
        await notificar_cuidador(reserva_id)
        await notificar_cliente(reserva_id)

    elif estado == "Rechazada":
        await actualizar_reserva(reserva_id, estado="fallida")
        await notificar_cliente_pago_fallido(reserva_id)

    elif estado == "Pendiente":
        # PSE o efectivo — esperar confirmación final
        await actualizar_reserva(reserva_id, estado="pendiente")

    return {"status": "ok"}

def registrar_cuidador_epayco(nombre, banco_codigo, numero_cuenta, tipo_cuenta):
    """
    banco_codigo: código bancario ePayco
    tipo_cuenta: "01" ahorros, "02" corriente
    """
    payload = {
        "P_CUST_ID_CLIENTE": P_CUST_ID,
        "P_KEY": P_KEY,
        "name": nombre,
        "bank": banco_codigo,
        "account": numero_cuenta,
        "type_account": tipo_cuenta
    }
    response = requests.post(
        "https://secure.epayco.co/restpagos/receiver/create",
        json=payload
    )
    return response.json()  # Retorna id_receiver para almacenar en DB

Consideraciones de seguridad

RiesgoMitigación
Exposición de P_KEYNunca enviar al frontend. Solo en variables de entorno del servidor.
Manipulación del monto desde el clienteEl monto y el split se calculan y firman en el backend.
Webhook falso (fraude)Siempre validar x_signature con SHA256 antes de procesar.
Double-spendingUsar x_ref_payco único + idempotencia en base de datos.
SQL injection en IDsUsar ORM o queries parametrizadas con el reserva_id.

Flujo técnico — secuencia de llamadas

GET /api/pagos/token?reserva={id} (DuniPets backend) └── Calcula split (20% DuniPets, 80% sitter) └── Obtiene id_receiver del sitter desde DB └── Firma la transacción con P_KEY └── Retorna datos de configuración al frontend 3. Frontend abre ePayco checkout con datos firmados 4. Usuario completa pago en checkout ePayco 5. ePayco → POST /api/webhook/epayco (DuniPets backend) └── Valida firma SHA256 └── Actualiza reserva en DB └── Dispara notificaciones (cliente + cuidador) 6. ePayco ejecuta split automáticamente └── 80% → cuenta bancaria / Daviplata del cuidador (1–3 días hábiles) └── 20% → cuenta principal DuniPets (1–3 días hábiles)
